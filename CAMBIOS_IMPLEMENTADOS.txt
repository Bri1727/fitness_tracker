7. DECISIONES TÉCNICAS Y JUSTIFICACIONES
-----------------------------------------

7.1. Debounce de 3 segundos con antirrebote
    - El stream de activity_recognition puede emitir cambios transitorios
      (ej. STILL -> WALKING -> STILL en pocos segundos)
    - El timer SOLO se reinicia cuando el tipo de actividad CAMBIA.
      Si el stream sigue emitiendo el mismo tipo, el timer sigue corriendo.
    - 3s es suficiente para filtrar fluctuaciones en emuladores sin sensores
      reales, y responde más rápido a cambios reales del usuario.
    - Antirrebote: se usa _pendingActivity como estado candidato. Solo se
      confirma y anuncia cuando el timer expira sin haber cambiado el tipo.

7.2. Umbral de caída > 35 m/s²
    - Magnitud normal en reposo: ~9.8 m/s² (gravedad)
    - Al caminar: ~10-15 m/s²
    - Al correr: ~15-25 m/s²
    - Una caída real: pico > 35 m/s² (impacto fuerte)
    - 35.0 evita falsos positivos con movimientos bruscos al correr

7.3. Cooldown de 15s en caídas
    - Una caída puede generar múltiples picos en fracciones de segundo
    - 15s evita múltiples diálogos por el mismo incidente

7.4. runForegroundService: true
    - Android 12+ requiere foreground service para Activity Recognition
    - El foreground service muestra una notificación permanente
    - Necesario para que el stream reciba eventos

7.5. Idioma del TTS
    - Intenta usar el idioma del dispositivo (locale)
    - Si no es español ni inglés, usa español por defecto
    - Verifica disponibilidad del idioma antes de asignarlo


8. FIX: DETECCIÓN DE ACTIVIDAD — OSCILACIÓN (19-Jun-2026)
---------------------------------------------------------

8.1. Problema: El estado oscilaba (stationary↔walking) al menor movimiento
    - El debounce anterior (ventana fija de 3s) solo miraba el ÚLTIMO tipo
      dentro de la ventana. Si el sensor alternaba tipos, bastaba un evento
      del otro tipo al final para cambiar el estado.
    - Con la ventana fija, un patrón STILL→WALKING→STILL en 3s terminaba
      anunciando "Te has detenido" aunque el usuario estuviera caminando.

8.2. Solución: Confirmación por eventos consecutivos (activity_service.dart)
    - Se eliminó el concepto de "ventana de tiempo" y _pendingActivity
    - Nuevo sistema: se requiere N eventos CONSECUTIVOS del mismo tipo
      (diferente al último anunciado) para confirmar un cambio
    - Cualquier evento que coincida con _lastAnnounced RESETEA la
      confirmación inmediatamente
    - requiredConsecutive = 2: se necesitan 2 eventos seguidos del nuevo
      tipo para cambiar de estado
    - confirmTimeoutSeconds = 20: timeout de seguridad por si el sensor
      deja de enviar datos

8.3. Trazas de ejemplo:
    Walking → ruido (oscilación):
      STILL(60)  → NEW confirming=stationary count=1
      WALKING(70)→ BACK to last → RESET (nunca llega a 2)
      → Sin anuncio ✓

    Walking → se detiene (estable):
      STILL(80)  → NEW confirming=stationary count=1
      STILL(75)  → CONFIRMED → "Te has detenido"
      → 2 eventos consecutivos confirman ✓

    Walking → RUNNING con ON_FOOT interrumpiendo:
      RUNNING(75)→ NEW confirming=running count=1
      ON_FOOT(80)→ mapped=walking==last → RESET
      RUNNING(70)→ NEW confirming=running count=1
      RUNNING(85)→ CONFIRMED → "Estás corriendo"
      → Se confirma, aunque con retraso por la interrupción ✓

8.4. Cambios en activity_service.dart:
    - Eliminado: _pendingActivity, _debounceTimer, _finalizeWindow
    - Nuevos: _confirmingType, _confirmingCount, _confirmTimer
    - Nuevo: _resetConfirm() (centraliza la lógica de reseteo)
    - requiredConsecutive = 2 (eventos consecutivos necesarios)
    - confirmTimeoutSeconds = 20 (timeout de seguridad)
    - Lógica de _onActivityEvent simplificada:
      1. SKIP si confianza < minConfidence
      2. RESET si mapped == _lastAnnounced (vuelta al estado actual)
      3. INCREMENTAR count si mapped == _confirmingType
         → Si count >= requiredConsecutive: CONFIRMAR
      4. SINO: nuevo confirmingType, count=1, iniciar timer
    - Se mantienen: minConfidence=40, filtro primera detección stationary,
      logging exhaustivo, ON_FOOT→walking


9. REWRITE: DATA SOURCE DEL ACELERÓMETRO (19-Jun-2026)
-------------------------------------------------------

9.1. Problema: Extremadamente sensible, pasos dobles/triples por pisada
    - El código nativo (Kotlin) hacía step detection y clasificación de
      actividad con umbrales mal sintonizados y sin anti-rebote
    - magnitude > 12 disparaba un paso, permitiendo múltiples incrementos
      por la vibración de una sola pisada
    - La clasificación por promedio móvil de 10 muestras reaccionaba a
      micro-movimientos, alternando "Quieto"↔"Caminando" constantemente

9.2. Solución: Toda la lógica migrada a Dart DataSource con 3 reglas

    9.2.1. Anti-Rebote de 300ms para PASOS (Debounce)
        - Variable _lastStepTime: solo se suma un paso si han pasado
          MÁS de 300ms desde el último paso
        - Una pisada genera múltiples picos de magnitud; el primero supera
          el umbral y suma el paso; los siguientes son ignorados por
          los próximos 300ms
        - Esto elimina por completo los pasos dobles/triples

    9.2.2. Ventanas de Tiempo de 3s para ESTADOS (Time Windows)
        - _lastWalkTime y _lastRunTime se actualizan CADA VEZ que
          la magnitud supera 13.0 o 17.0 respectivamente
        - El estado se determina NO por el último evento, sino por
          cuándo fue la última vez que se superó cada umbral:
          * Si hace < 3s desde _lastRunTime → CORRIENDO
          * Si hace < 3s desde _lastWalkTime → CAMINANDO
          * Si hace >= 3s desde ambos → QUIETO
        - Esto crea un "cojín de 3 segundos": el usuario puede hacer
          una pausa de 1-2 segundos y la app sigue reportando el
          estado anterior, evitando falsos "Quieto"

    9.2.3. Umbrales ajustados
        - Paso / Caminata: magnitud > 13.0 m/s²
        - Correr: magnitud > 17.0 m/s²
        - Caída: magnitud > 35.0 m/s²

9.3. Archivos modificados:

    android/.../MainActivity.kt:
        - Eliminado: moving average (magnitudeHistory), step detection
          (stepCount, lastMagnitude), actividad (lastActivityType,
          activityConfidence), throttling (sampleCount)
        - Simplificado a: recibir cada muestra, calcular magnitud
          bruta sqrt(x²+y²+z²), enviarla inmediatamente vía
          events?.success(magnitude) como Double
        - MethodChannel handlers ("start"/"stop"/"reset") ahora
          son no-ops (el estado se maneja en Dart)

    lib/features/auth/data/datasources/acelerometer_datasource.dart:
        - Reescripto completamente: recibe magnitud cruda del canal,
          aplica debounce de 300ms para pasos, ventanas de 3s para
          estado, emite StepData con stepCount y activityType
          computados localmente
        - Nuevos: _lastStepTime, _lastWalkTime, _lastRunTime,
          _processReading(), _resolveState()
        - stepStream getter ahora retorna _controller.stream (broadcast)

    lib/core/services/fall_detection_service.dart:
        - Umbral de caída: 25.0 → 35.0 m/s² (regla 4 del usuario)

9.4. Traza de ejemplo (pisada única con rebotes):
    t=0ms  magnitud=14.2 → >13.0, diff _lastStepTime = ∞ > 300ms → PASO+1 ✓
    t=100ms magnitud=12.8 → >13.0? NO → no hace nada
    t=200ms magnitud=13.5 → >13.0, diff=200ms < 300ms → ignorado ✓
    t=400ms magnitud=15.1 → >13.0, diff=400ms > 300ms → PASO+1 ✓
    → 2 pisadas reales, 2 pasos (sin dobles ni triples)

9.5. Traza de ejemplo (caminar → pausa 2s → caminar):
    t=0  caminando, magnitud=14 → _lastWalkTime=t0, estado=walking
    t=1s pausa, magnitud=9.8   → no supera 13.0, _lastWalkTime=t0
    t=2s aún en pausa           → diff desde t0 = 2s < 3s → estado=walking ✓
    t=3s aún en pausa           → diff desde t0 = 3s >= 3s → estado=stationary
    t=4s reanuda, magnitud=14   → _lastWalkTime=t4, estado=walking
    → La pausa de 2s no genera anuncio falso de "Quieto"


10. FIX: TTS NO HABLA (19-Jun-2026)
------------------------------------

10.1. Problema: El sistema detecta cambios de actividad pero el TTS
      no reproduce los mensajes de voz.

10.2. Causas diagnosticadas:

      a) init() usaba isLanguageAvailable() que es poco fiable en Android:
         si devolvía false para es-ES Y en-US, igualmente seteaba es-ES
         (no disponible) y marcaba _initialized = true, pero el motor
         no podía sintetizar ese idioma.

      b) No se verificaba el resultado de setLanguage(): si fallaba,
         _initialized se marcaba igual como true, y speak() llamaba
         al motor con un idioma que no funciona → silencio.

      c) No había test de verificación post-init: el monitor widget
         asumía que TTS funcionaba sin confirmarlo.

10.3. Cambios en tts_service.dart:

      - Eliminado: isLanguageAvailable() (no confiable en Android)
      - Nuevo: lista de candidatos [es-ES, es-MX, en-US, en-GB]
      - setLanguage() se prueba secuencialmente con cada candidato
        hasta que uno funciona (try/catch por cada intento)
      - Si ningún idioma funciona → _initialized = false, log claro
      - Nuevo método test(): intenta sintetizar "ok" y retorna bool
      - Nuevo getter isInitialized

10.4. Cambios en activity_monitor_widget.dart:

      - init() ahora llama _tts.test() después de _tts.init()
      - Solo habla el mensaje de bienvenida si test() es OK
      - El listener de activityStream verifica ttsOk antes de speak()
      - El listener de fallStream también verifica ttsOk
      - Logs con prefijo [Monitor] para facilitar depuración
